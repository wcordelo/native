# Platform Support

zero-native exposes one runtime model across desktop hosts, but native affordances land per platform and per web engine. Unsupported operations return explicit runtime errors instead of pretending that every host can do the same thing.

## Native Surfaces

The macOS, Linux, and Windows system-WebView hosts support `toolbar`, `titlebar_accessory`, `sidebar`, `statusbar`, `button`, `checkbox`, `toggle`, `text_field`, `search_field`, `label`, `spacer`, and `progress_indicator` native view kinds.

`ViewKind.webview` is the compatibility path for WebView-backed views. Existing `runtime.createWebView(...)` and `window.zero.webviews.*` APIs remain available.

## Mobile And GPU

iOS, Android, and `gpu_surface` are not part of the current supported desktop host matrix. The public API already reserves the native view vocabulary needed for those surfaces, but the shipped support today is desktop-first.
